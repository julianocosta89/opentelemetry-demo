# Run the demo on Kubernetes

`skaffold` builds every service from this fork's de-oteled source and deploys it to a
local cluster. (Don't `kubectl apply` the manifest directly — see [Background](#background).)

## 1. One-time setup

A local cluster, the `buildx` plugin, and a Docker Hub mirror. For **colima + minikube**
(no Docker Desktop licence needed):

```bash
# buildx plugin (colima's docker doesn't bundle it)
brew install docker-buildx
mkdir -p ~/.docker/cli-plugins
ln -sfn "$(brew --prefix)/opt/docker-buildx/bin/docker-buildx" ~/.docker/cli-plugins/docker-buildx

# cluster, with a Docker Hub pull-through mirror
colima start --cpus 4 --memory 8
minikube start --driver=docker --registry-mirror=https://mirror.gcr.io --cpus 4 --memory 6g
```

`.env` already sets `BASE_IMAGE_MIRROR=mirror.gcr.io`. Together with the `--registry-mirror`
above, that keeps builds off Docker Hub entirely (no `docker login` needed). Docker Desktop
and kind also work — see [Other clusters](#other-clusters).

## 2. Run it

```bash
skaffold dev      # build + deploy + live-reload; Ctrl-C tears it down
# or
skaffold run      # build + deploy once, leave it running
```

First build is slow (every service is compiled); later rebuilds are incremental. Open the
storefront at <http://localhost:8080>.

## 3. Check it's working

```bash
kubectl get pods -n otel-demo                              # all Running
curl -s -o /dev/null -w '%{http_code}\n' localhost:8080    # 200
```

## Adding or changing services — you don't touch `skaffold.yaml`

The build artifacts in `skaffold.yaml` are **regenerated by the sync pipeline on every
sync** — `sync/resolvers/k8s.py::ensure_skaffold_artifacts`. It derives the artifact list
from the k8s manifest: every `<prefix>/<service>-no-otel` image the manifest deploys gets a
build artifact, with its Dockerfile path taken from the compose file's
`services.<name>.build.dockerfile`. So:

- **A service is built and deployed iff it has a Deployment in the manifest.** When upstream
  adds a service that ships in the Helm/k8s manifest, the next sync covers it automatically —
  no manual `skaffold.yaml` edit.
- **Optional, compose-only services aren't deployed here.** `agent`, `chatbot`, and `mcp` are
  upstream's optional LLM features, shipped only via the layered `compose.agent.yaml` (not the
  k8s manifest), so they're not built or deployed by skaffold. Run them with docker-compose
  instead: `docker compose -f compose.yaml -f compose.agent.yaml up`.
- **Don't hand-edit the `artifacts:` list** — your edit is overwritten on the next sync. If a
  service should be built, it belongs in the manifest; the pipeline does the rest.

If a manifest service has no buildable Dockerfile, the pipeline leaves it out and flags it, and
the deploy-consistency gate (`tests/test_deploy_consistency.py`, part of `pytest -m fast`)
fails the sync → NEEDS-REVIEW, rather than producing a broken deploy.

## Troubleshooting

| Symptom | Fix |
|---|---|
| `TOOMANYREQUESTS` pulling a base image | Docker Hub anonymous limit — do **both** mirror steps in setup (`--registry-mirror` *and* `BASE_IMAGE_MIRROR`), or `docker login`. See [Docker Hub rate limits](#docker-hub-rate-limits). |
| `BuildKit is enabled but the buildx component is missing` | Install buildx (setup step 1). |
| `ErrImagePull` / `ImagePullBackOff` on a `*-no-otel` image | skaffold built into a daemon the cluster can't see — confirm `kubectl config current-context` is `minikube`/`kind`/`docker-desktop`, else use the `registry` profile below. |
| `open …/Dockerfile: no such file` during build | A skaffold artifact points at a missing Dockerfile. This should never reach you — the deploy-consistency gate catches it during sync. If you hit it on a hand-edited tree, run a sync (or `python -c "import sys;sys.path.insert(0,'sync');from resolvers import k8s;print(k8s.ensure_skaffold_artifacts('.'))"`) to reconcile. |
| Pods `Pending` / `OOMKilled` | Cluster too small. `minikube stop && minikube start --memory 10g --cpus 6`. |

## Other clusters

- **Docker Desktop:** `skaffold run` — shared daemon, nothing extra.
- **kind:** `kind create cluster && skaffold run`.
- **Remote / colima k3s / anything skaffold can't load into:** push to a registry —
  `skaffold run -p registry --default-repo=<your-registry>`.

## Background

Details you don't need to run it, but help when something looks off or you're changing how
it's built.

### Why not `kubectl apply`?
The manifest is generated from upstream and originally points at **prebuilt images that still
contain OpenTelemetry**. The sync pipeline rewrites those refs to
`<NO_OTEL_IMAGE_PREFIX>/<service>-no-otel` (default prefix `ghcr.io/julianocosta89`, set in
`.env`/`sync/config.py`) so skaffold matches each to an artifact it builds from de-oteled
source. `tests/test_local_images.py` guards that no prebuilt `ghcr.io/open-telemetry/demo:`
image ever reaches the manifest.

### How skaffold artifacts stay in sync
`ensure_skaffold_artifacts` (run every sync) reconciles `skaffold.yaml`'s `artifacts:` to
exactly the manifest's de-oteled services: existing artifacts are kept verbatim, missing ones
added, orphans removed. The service→Dockerfile mapping comes from the compose file
(`services.<name>.build.dockerfile`, `${VAR}` resolved via `.env`) — authoritative because the
compose service name is correct where an env-var stem isn't (e.g. `fraud-detection`'s var is
`FRAUD_DOCKERFILE`), and it captures non-uniform paths like `cart → src/cart/src/Dockerfile`.
postgres is excluded — it runs the stock image with `init.sql` from a ConfigMap, not a build
(`ensure_postgres_deployable`). The deploy-consistency gate enforces the result.

### buildx requirement
The Dockerfiles use `FROM --platform=${BUILDPLATFORM} …`, a BuildKit-only feature, so
`skaffold.yaml` keeps `useBuildkit: true` and a BuildKit builder (`buildx`) is required.

### Docker Hub rate limits
Anonymous pulls are capped at 100 / 6h per IP. Two independent paths hit Hub, so the mirror
fix has two halves: image-layer pulls → minikube's daemon `--registry-mirror` (set at cluster
*creation* only); base-image metadata → skaffold inspects each base via a direct
`index.docker.io` call that ignores the daemon mirror, so `BASE_IMAGE_MIRROR` makes the
pipeline rewrite every `FROM docker.io/...` to the mirror. Set `BASE_IMAGE_MIRROR=` empty to
disable.

### Conventions
- Namespace `otel-demo` (manifest `Namespace` + skaffold `defaultNamespace` agree).
- `build.local.push: false` + `concurrency: 4`; the `registry` profile flips `push: true`.
